### Demo05 - Edge Cases run away dynamic SQL

### Overview

While the root cause is some sort of application logic failure/bug, I would like the database engine to ensure one bad query distrupt other users and/or take up too much resources.

Below are some simple scripts which are designed to replica scenarios with extreme inserts first with 2000 (2K) rows/bind variables, then 4K, then 40K, then 4000K.

Historically I've seen more problems around smaller database VMs (4G and 8G) and in particular for DBAAS. 

In DBAAS more likely to silently failover with no details of the root application logic failure/bug i.e. your run away resource usage is likely to recur.

TODO - the scripts below have the number of bind variables hard coded (2K, 4K, 40K, 4000K) this should be parameterised.
